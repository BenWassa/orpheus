**Status**: Phase 2 (Friend Testing)  
**Timeline**: MVP target  
**Owner**: Benjamin Haddon  
**Last Updated**: 2026-05-27

---

## Executive Brief

Build a React/TypeScript front-end for **Orpheus**, a personal music listening analysis tool that surfaces emotional and thematic patterns in Spotify listening history. The backend is a Python CLI that generates JSON reports; the frontend consumes those reports and presents insights through an interactive, single-page web application.

**The core insight**: What you play repeatedly is data about what you are paying attention to internally. The frontend's job is to make that data feel like a meaningful mirror, not a spreadsheet.

---

## Mission

Make the user's emotional and thematic listening patterns **visible, navigable, and reflective**. Transform raw analysis data into an experience that feels:

- **Observational** — not judgmental  
- **Intuitive** — patterns should be immediately readable  
- **Explorable** — let users dig deeper into connections  
- **Minimal** — read-in-five-minutes narrative, not overwhelming dashboards  

The interface should feel like **opening a letter about yourself**, not a corporate analytics tool.

## Project Scope

### In Scope (MVP)

1. **Report Upload & Parsing**  
   - Users drag-and-drop or select a JSON report generated by the backend  
   - Frontend validates schema and normalizes the report into the frontend contract  
   - MVP supports the current backend report shape plus the target report shape in `02_Output_Schema.md`

2. **Summary View (Landing Page)**  
   - Headline: top emotional categories and themes (state window)  
   - Quick comparison to trait baseline  
   - Key shifts / rising or falling patterns  
   - Narrative-style summary composed from backend shift/trend/co-occurrence narratives  

3. **Emotion Axis Visualization**  
   - Interactive 2D plot (Valence × Arousal) with Depth as size/glyph  
   - State (current) vs Trait (baseline) comparison  
   - Hover/click for cluster or category detail when report data supports it  

4. **Theme Axis Exploration**  
   - 8 life themes presented in visual or textual format  
   - Prevalence labels (dominant, high, moderate, present)  
   - Click to inspect theme description, prevalence, state-vs-trait delta, and related shifts  

5. **Cluster & Evidence View**  
   - List clusters with labels, AVD centroid, dominant emotions/themes, and track count  
   - Show top tracks only when backend report includes them  
   - Avoid per-track score explanations until track-level report data is emitted  

6. **Trend Timeline**  
   - MVP displays backend trend events: category, axis, direction, magnitude, narrative  
   - Full weekly line chart requires a future time-series report contract  

7. **Co-occurrence Visualization**  
   - Show which theme + emotion pairs appear together  
   - MVP uses a ranked list or compact matrix based on current backend `pair` output  

8. **Data Export**  
   - Download normalized report JSON  
   - Share-safe export excludes raw listening history and any track-level data not already in the report  

### Out of Scope (Phase 2+)

- **Multi-file comparison** (comparing multiple report dates)  
- **Social sharing** or public profiles  
- **Live Spotify API integration** (backend handles this; frontend receives static report)  
- **Mobile-optimized PWA** (desktop first; mobile adaptation deferred)  
- **Real-time analysis** (batch processing only)  
- **Multi-language support** (English only for MVP)  
- **Dark mode** (consider but not required)  
- **Full track drill-down with album art and score explanations** (requires track-level report data)  
- **Weekly multi-series trend chart** (requires weekly time-series output, not just trend events)  
- **Generated prose narrative as a required backend field** (frontend can compose MVP summary copy)

## User Personas & Stories

### Persona 1: Self-Reflector (Primary)
**Goal**: Understand their own listening patterns  
**Behavior**: Uploads report, reads summary, explores emotion/theme clusters, saves insights  
**Pain point**: Wants quick sense-making (5–10 min), not overwhelming detail  

> *As a self-reflector, I want to upload my Orpheus report and immediately see which emotions and themes dominate my listening, so I can ask myself "Is this what I'm working through right now?"*

### Persona 2: Curious Explorer
**Goal**: Dig deeper into patterns  
**Behavior**: Uploads report, explores trends, inspects category evidence, compares clusters  
**Pain point**: Wants richness but still intuitive navigation  

> *As an explorer, I want to click on "Tenderness" and see whether it is elevated, which themes it co-occurs with, and which clusters it belongs to, so I can understand the pattern without needing a spreadsheet.*

### Persona 3: Data Skeptic
**Goal**: Validate the analysis feels accurate  
**Behavior**: Spot-checks shifts, trend narratives, cluster weights, and co-occurrence logic  
**Pain point**: Needs transparency to trust the system  

> *As a skeptic, I want to inspect the numbers behind a claim, so I can evaluate whether the analysis is real or noise.*

## Key Functional Requirements

### FRQ-1: Report Upload & Validation
- **Input**: JSON file (Orpheus backend report schema)  
- **Output**: Validated data structure in application state  
- **Behavior**:  
  - Drag-and-drop or file input  
  - Real-time schema validation (show errors clearly)  
  - Reject malformed reports with helpful messages  
  - Support multiple reports in sequence (clear/upload new)  

### FRQ-2: Summary Dashboard
- **Display**:  
  - Headline: "Your listening clusters around [top emotion] & [top theme]"  
  - State-vs-trait comparison (badges: rising, stable, falling)  
  - Key insights composed from backend shifts, trends, co-occurrences, and optional narrative  
  - Visual preview of emotion/theme distribution  
- **Behavior**:  
  - Auto-generate on report upload  
  - Clickable elements route to category, cluster, or evidence detail views  
  - Mobile-friendly scrollable summary  

### FRQ-3: Emotion Axis Visualization
- **Visualization**: 2D scatter plot (Valence × Arousal)  
- **Encoding**:  
  - X-axis: Valence (−1 to +1)  
  - Y-axis: Arousal (−1 to +1)  
  - Bubble/circle size: Depth  
  - Color: State vs Trait (or category)  
  - Opacity: Cluster membership strength  
- **Interactivity**:  
  - Hover: Show emotion name, prevalence, and score when available  
  - Click: Open category detail with related shifts, trends, and clusters  
  - Toggle: State ↔ Trait  
  - Filter: By axis/category once numeric distributions are available  
- **Reference**: See `01_Emotion_Taxonomy.md` for category definitions  

### FRQ-4: Theme Axis Display
- **Display**: 8 life themes with prevalence labels  
- **Format options**:  
  - Horizontal bar chart (length = prevalence)  
  - Ordered list with visual badges  
  - Grid cards with hover tooltips  
- **Interactivity**:  
  - Click theme → show category detail, prevalence, related shifts, and related co-occurrences  
  - Filter by minimum prevalence  
  - Show track evidence only when the report includes track-level data  
- **Reference**: See `01_Emotion_Taxonomy.md` for theme definitions  

### FRQ-5: Cluster & Evidence Detail
- **Display**:  
  - Cluster label  
  - AVD centroid  
  - Dominant emotions and themes  
  - Share of listening  
  - Track count  
- **Filtering**:  
  - By emotion category  
  - By theme category  
  - By cluster label or strength  
- **Details panel**:  
  - Explain related shifts, trends, and co-occurrences  
  - Show top tracks only if backend report includes them  
  - Defer album art, lyrics snippets, and score breakdowns until the backend emits track-level details  

### FRQ-6: Trend Analysis
- **Display**: Recent movement list from backend trend events  
- **Rows**: Category, axis, direction, magnitude, narrative  
- **Interactivity**:  
  - Filter emotion/theme axes  
  - Filter notable movement  
  - Expand a row for supporting narrative  
  - Defer 12-week line chart until backend emits weekly series data  

### FRQ-7: Co-occurrence Matrix
- **Display**: Theme × Emotion pairs that appear together  
- **Format options**:  
  - Heatmap (intensity = co-occurrence frequency)  
  - Network graph (nodes = categories, edges = co-occurrences)  
  - Table with counts  
- **Interactivity**:  
  - Click cell/edge → show observed count, expected count, strength, and narrative  
  - Filter by minimum co-occurrence strength  
  - Highlight related categories  

### FRQ-8: Export & Share
- **Formats**:  
  - JSON (normalized report)  
  - CSV and PNG deferred  
- **Safety**:  
  - Never export individual listening history  
  - Only export aggregated patterns + narrative

## Non-Functional Requirements

### Performance
- **Page load**: <2s on broadband (static file parsing, no network calls)  
- **Chart render**: <500ms for VA scatter + all theme bars  
- **Drill-down interactions**: <200ms for filter/toggle operations  
- **Memory**: Should handle reports up to 50MB JSON (typical report: 2–5MB)  

### Accessibility
- **WCAG 2.1 AA compliant** minimum  
- **Color**: Emotion charts must be colorblind-safe (avoid red/green alone)  
- **Keyboard navigation**: All interactive elements accessible via Tab  
- **Screen reader**: Chart descriptions, alt text for images  
- **Focus indicators**: Visible on all interactive elements  

### Browser Support
- **Chrome/Edge**: Latest 2 versions  
- **Firefox**: Latest 2 versions  
- **Safari**: Latest 2 versions  
- **Mobile browsers** (future): iOS Safari, Chrome Mobile  

### Security
- **File parsing**: Validate JSON schema strictly; reject unknown keys with warning  
- **No network calls** for report loading (front-end only, no analytics tracking)  
- **No local storage** of raw listening history (computed patterns only, if any)

## Data Contract Decision

The frontend plan uses a hybrid contract strategy:

- **Current backend contract (`ReportV0`)**: The Python CLI currently emits `generated_at`, `model_version`, `windows.state`, `windows.trait`, `shifts`, `trends`, `co_occurrences`, `clusters`, and `safety_flags`.
- **Target frontend contract (`ReportV1`)**: The frontend should normalize reports into a stable shape with `metadata`, `state`, `trait`, `shifts`, `trends`, `co_occurrences`, `clusters`, and optional `narrative`.
- **Implementation rule**: Frontend components consume only normalized `ReportV1`. Upload/parsing code owns compatibility with `ReportV0`.

This keeps the frontend build aligned with backend reality without lowering the product direction.

### Current Backend Shape

The backend currently produces a report shaped like this:

```json
{
  "generated_at": "ISO8601",
  "model_version": "string",
  "windows": {
    "state": {
      "top_emotions": [{ "category": "nostalgia_longing", "prevalence": "dominant" }],
      "top_themes": [{ "category": "heartbreak_loss", "prevalence": "high" }],
      "depth_label": "engaged",
      "top_artists": [],
      "top_tracks": []
    },
    "trait": {
      "top_emotions": [],
      "top_themes": [],
      "depth_label": "engaged",
      "top_artists": [],
      "top_tracks": []
    }
  },
  "shifts": [],
  "trends": [],
  "co_occurrences": [],
  "clusters": [],
  "safety_flags": []
}
```

### Target Frontend Shape

```json
{
  "metadata": {
    "generated_at": "ISO8601",
    "model_version": "string",
    "schema_version": "1.0"
  },
  "state": {
    "emotion": { "joyful_activation": 0.24, ... },
    "theme": { "interpersonal_devotion": 0.18, ... },

  },

    { "category": "nostalgia_longing", "axis": "emotion", "direction": "rising", "magnitude": "notable" },
```

## Technical Stack Recommendation

### Frontend Framework
- **React 18+** (TypeScript)  
- **Vite** for build tooling (fast dev, optimized bundles)  

### State Management
- **Zustand** (simple, lightweight client state for uploaded report and UI filters)  
- **No TanStack Query for MVP** (there is no remote server state; add it only if a backend API is introduced)  

### Visualization
- **Recharts** (batteries-included charting, React-native)  
  *OR*  
- **Plotly.js** (more control, larger bundle)  
  *OR*  
- **Visx** (low-level building blocks, more flexibility)  

### UI Component Library
- **Radix UI** (unstyled, accessible primitives)  
- **Tailwind CSS** (utility-first styling)  
- **Shadcn/ui** (pre-built Radix + Tailwind components)  

### File Handling
- **React-Dropzone** (drag-and-drop)  
- **Native File API** (reading + parsing)  

### Data Validation
- **Zod** (TypeScript-first schema validation)  

### Testing
- **Vitest** (fast unit tests)  
- **React Testing Library** (component tests)  
- **Playwright** (e2e tests)  

### Deployment
- **Vercel** (zero-config, optimized for Next.js or Vite)  
  *OR*  
- **Netlify** (alternative, similar)  
  *OR*  
- **Static hosting** (S3 + CloudFront, if keeping costs low)

## Design Direction

### Visual Aesthetic
- **Tone**: Introspective, calm, human-centered  
- **Palette**: Muted, analogous colors (avoid aggressive reds/blues)  
- **Typography**: Clean serif headings (Crimson Text, Fraunces) + sans body (Inter, System)  
- **Spacing**: Generous; embrace whitespace  
- **Motion**: Subtle transitions (easing functions, <300ms), no gratuitous animation  

### Layout Patterns
- **Hero summary** (top): Large, bold headline + state-vs-trait snapshot  
- **Main content area** (mid): 3-column or single-column responsive  
  - Left: Emotion chart (VA scatter)  
  - Center: Theme prevalence  
  - Right: Recent trends OR co-occurrence heatmap  
- **Drill-down** (modal or side panel): category evidence, cluster summaries, co-occurrence details  
- **Footer**: Export, metadata, links to docs  

### Emotional Design Goals
- **Visual feedback**: Charts should feel responsive and alive (smooth transitions on data changes)  
- **Narrative integration**: Blend text + visuals so insights feel like a story, not a dashboard  
- **White space**: Let breathing room reduce cognitive load  
- **Consistent iconography**: Use icons to reinforce emotion/theme categories

## Risks & Mitigations

| Risk | Probability | Impact | Mitigation |
|---|---|---|---|
| Schema changes mid-project | Medium | High | Agree on schema freeze before frontend work; use versioning if changes occur |
| Large JSON files cause performance issues | Medium | Medium | Test with 50MB+ files early; consider streaming or chunking if needed |
| Chart library doesn't support needed interactions | Low | Medium | Spike on Recharts/Visx interactivity early; prototype touch-friendly zoom |
| Accessibility requirements missed | Medium | Medium | Integrate axe-core into CI; conduct audit at end of Phase 4 |
| Unclear data semantics from backend | Low | Medium | Schedule weekly syncs; request clear documentation (see context files) |
